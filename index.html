1. System description

    1.1 Basic Information about the System

    The Car Rental Management System is comprehensive software designed to manage all aspects of a car rental business, integrating internal processes with customer interactions. The system covers vehicle reservations, rentals, returns, fleet maintenance, staff management, and real-time vehicle status monitoring. It also includes integrated billing and payment handling, enabling easy transaction management. With this integration, the system provides centralized access to information, enabling efficient operations.

    1.2 Description of the Modeled Reality

  • Vehicle: Each vehicle in the fleet has attributes such as make, model, production year, license plate, vehicle type (e.g., SUV, sedan, hatchback), mileage, technical condition, and a unique identifier. Vehicles can be rented, reserved, or flagged for maintenance based on their condition.
  • Client: Each client profile includes personal details, rental history, reservations, billing information, and any restrictions based on prior rental behavior.
  • Employee: Employees of the car rental company are divided into roles such as Customer Service Representatives, Maintenance Technicians, Managers, and Administrators.
  • Shift and Schedule Management: Managers create shift schedules and assign specific responsibilities to employees. The system records shift start and end times, assigned tasks, and employee availability.
  • Billing and Payments: The system calculates rental costs based on vehicle type, rental duration, mileage, additional services, and taxes. Payment options include credit cards, debit cards, and electronic payments. Late fees, penalties, or discounts are automatically applied based on company policy and client rental history.
  • Operations: Core operations include vehicle reservation, rental, return, cancelation, maintenance scheduling, vehicle availability updates, and service record tracking.

    1.3 User Types

  • Clients: Can browse available vehicles, make reservations, rent vehicles, manage their rental history, and process payments.
  • Customer Service Representatives: Assist with client reservations, process rentals, handle returns, and manage payments. They can make reservations directly at the front desk for clients who arrive in person.
  • Maintenance Technicians: Track repair statuses, log service information, and update vehicle readiness.
  • Managers: Responsible for employee oversight, approving staffing decisions, managing shift schedules, and assigning tasks. They also handle escalated customer issues and approve specific client requests.
  • Administrators: Manage system configurations, user roles, and permissions. They monitor security, oversee data access, and handle account setups for new employees.

    1.4 System Functionality

  • Vehicle Reservation: A client (or a Customer Service Representative on their behalf) selects a vehicle, chooses rental dates, and confirms the reservation. The system then marks the vehicle as reserved for the specified period.
  • Vehicle Rental: When the client arrives, the Customer Service Representative verifies their profile, confirms the reservation, and initiates the rental. The vehicle status is then updated as rented.
  • Vehicle Return: Upon completion, the vehicle is returned, and the status is updated to available. The representative logs any extra charges, fuel levels, and vehicle condition.
  • Reservation Cancelation: A client can cancel reservation before picking up the vehicle, extra charges will be added depending of time remaining to pick-up
  • Billing and Payments: At the end of each rental, the system generates an invoice based on the rental agreement. Charges include base fees, taxes, and any additional costs, such as mileage overage or late fees. Clients can pay directly through the system or at the front desk. Payment status and history are updated in the client profile, and receipts are issued electronically.
  • Vehicle Maintenance: Maintenance Technicians update service records for each vehicle, flagging those under maintenance as unavailable.
  • Shift and Schedule Management: Managers manage employee schedules, assigning specific shifts and tasks. The system logs shift start and end times, task completion, and employee availability. It also provides reports on staffing and usage, helping Managers to optimize resource allocation.
  • User Management and Role Assignment: Administrators assign roles, set permissions, and handle employee onboarding.

    1.5 Non-Functional Requirements

  • Performance: The system supports real-time bookings and fleet updates, even under high transaction volumes.
  • User Interface: The interface should be user-friendly, mobile-responsive, and tailored to each user role, allowing easy navigation to relevant functions.
  • Security and Access Control: Security protocols protect client data and restrict access based on user roles. System logs track all actions for accountability.

2. Dictionary

  • Vehicle: car or similar mode of transport within the rental company’s fleet. Each vehicle has specific details, such as make, model, year, license plate, mileage, and condition status, which are tracked in the system.
  • Client: An individual or organization renting a vehicle from the company. Client profiles include personal information, rental history, billing records, and any relevant restrictions.
  • Fleet: The complete collection of vehicles owned by the rental company and available for rental. The fleet is managed within the system, tracking availability, maintenance schedules, and rental status for each vehicle.
  • Reservation: A client’s booking of a vehicle for a specific period. The reservation process includes selecting a vehicle, choosing rental dates, and confirming the booking, after which the vehicle status is updated as reserved.
  • Rental: The active usage period of a vehicle by a client, initiated at the start of the rental contract. During this period, the vehicle is marked as unavailable for other reservations until returned.
  • Return: The process by which a client returns a rented vehicle, allowing for an inspection and status update in the system. Return procedures also involve recording fuel levels, any damages, and finalizing billing.
  • Customer Service Representative: An employee who assists clients with reservations, rental processing, returns, and payments. They interact directly with clients, managing reservations both online and in person.
  • Maintenance Technicians: An employee who is responsible for maintenance of the fleet, ensures vehicle safety and performance, flaggs in the system when vehicles are unavailable for rental due to service needs.
  • Manager: An employee responsible for overseeing other staff, managing shifts, assigning tasks, and handling escalated issues. Managers have permissions to approve staffing decisions, assign specific employee roles, and optimize resource allocation within the system.
  • Administrator: An employee responsible for managing system settings, user permissions, security, and data access. Administrators oversee the integrity of the system, handle role assignments, and manage data control policies.
  • Staff: All employees within the rental company, including Customer Service Representatives, Maintenance Technicians, Managers, and Administrators. Each staff member has specific permissions and responsibilities within the system.
  • Billing: The process of calculating and invoicing costs associated with a rental, including base fees, additional charges, taxes, and any penalties. Billing information is associated with each client profile and updated upon rental completion.
  • Payment: The transaction that completes the rental process, with clients settling outstanding charges. Payments can be made directly through the system or at the front desk, and are logged in client billing records.
  • User Interface: The visual and interactive components of the system that allow users to navigate, input, and access information. The interface is tailored to different user roles, providing a clear and efficient experience for all employees and clients.

8. Design Patterns

8.1 Prototype

Karol Wójcik

Problem

System needs efficient way to handle creating multiple instances of similar cars for various rental requests.
For example, if we frequently rent out a specific model with the same configuration (e.g., engine type, color, and features), creating each instance from scratch can be resource-intensive and error-prone.
Additionally, as the system evolves, there may be a need to customize these cars slightly (e.g., adding specific features for a particular customer).

Solution

To address this problem, the system requires a mechanism to:

*Quickly create copies of existing car objects without repeating initialization logic.
*Ensure that the copied objects are independent of the original and can be modified without side effects.
*Reduce the cost of creating objects by reusing existing configurations as prototypes.

The Prototype design pattern provides a way to create new objects by copying an existing object, known as the prototype. This approach is particularly useful when creating resource-heavy objects or when object configurations is complex.

8.2 Abstract

Jakub Sikora

Problem

The system needs an efficient way to handle creating multiple instances of similar employees for various roles (e.g., Customer Service Representatives, Maintenance Technicians, Managers, and Administrators) for different tasks or shifts. These employees share common attributes (e.g., name, surname, role, tasks) and behaviors, making it inefficient and error-prone to recreate them from scratch for every instance. Additionally, as the system evolves, there may be a need to slightly customize these employee instances (e.g., adding specific tasks or modifying attributes for particular shifts or duties). Without a centralized structure to manage shared functionality and attributes, this process could become repetitive and prone to inconsistencies.

Solution

To address this problem, the system requires a mechanism to: Centralize common employee attributes and behaviors in a shared structure to reduce duplication and errors. Quickly create copies of existing employee objects without repeating initialization logic. Ensure that the copied employee objects are independent of the original and can be modified without side effects (e.g., adjusting tasks or assignments for specific shifts). Using an abstract class to implement the Prototype design pattern solves these challenges effectively. The abstract class defines common properties (name, surname, role, tasks) and shared behaviors (addTask, displayInfo, performTask). Subclasses for each role (e.g., Customer Service Representatives, Technicians) can inherit this shared functionality while implementing their specific cloning logic through the abstract clone() method. This approach simplifies object creation, ensures reusability of employee configurations, and allows for efficient cloning and modification of employee roles without the overhead of re-initializing common attributes.

            import java.util.*;

abstract class Employee implements Cloneable {

    protected String surname;
    protected String role;

        this.name = name;
        this.surname = surname;
        this.role = role;
        this.tasks = new ArrayList<>();

    public void addTask(String task) {
        tasks.add(task);

    }

    public abstract Employee clone();

    public void performTask(String task) {
        System.out.println(role + " " + name + " is performing task: " + task);
    }
}

9. Structural Patterns

9.1 Composite

Karol Wójcik

Problem

We need to manage and represent collections of cars as a single object. For instance, we may group cars into fleets based on their type, purpose, or availability. These fleets need to be treated uniformly whether they represent a single car or a collection of cars. Additionally, it should be possible to compose fleets of fleets to represent larger groupings, such as all cars available for a particular event.

Solution

The system requires a mechanism to:

*Represent both individual cars and groups of cars uniformly.
*Allow composition of groups.
*Simplify operations such as displaying fleet details or performing actions on all cars within a group.

The Composite design pattern is used to treat individual objects and compositions of objects uniformly. This pattern allows a tree structure where objects can either be individual components (leaves) or composites (containers).

9.2 Bridge

Jakub Sikora

Problem

In a user management system with different types of users (e.g., technicians, managers, administrators), it is necessary to provide flexible management of access to specific resources or functions. Users should have different levels of permissions (e.g., updating car status, creating schedules), and their permissions should be defined independently of their role in the system. It is essential to separate user logic from permission logic to enable easy system expansion and maintain code clarity.

Solution

The Bridge design pattern allows separating user logic (e.g., CustomerServiceRepresentative, Manager) from their assigned permissions (Permission). This enables independent development of the user hierarchy and the permission hierarchy, avoiding complex class combinations. Users delegate the execution of operations to their assigned permissions.

10.1 Observer

Karol Wójcik

Problem

System needs to notify various modules (e.g., availability tracking, customer notifications, pricing updates) whenever the status of a car changes (e.g., when it is rented, returned, or marked for maintenance). These modules act as observers that must stay synchronized with the car's status. For example:

*When a car's status changes to "rented," customers on the waiting list must be notified.
*When a car's status changes to "maintenance," the availability tracking system should mark it unavailable.
*The pricing system may adjust rental prices dynamically based on demand and availability.

Solution

The Observer design pattern defines a one-to-many dependency between objects so that when one object (the Subject) changes its state, all its dependents (Observers) are notified and updated automatically.

Class diagram

@startuml interface Observer { + update(String status) } class Car { - status : String - observers : List + addObserver(Observer) + removeObserver(Observer) + setStatus(String) + notifyObservers() } class CustomerNotification implements Observer { + update(String status) } class AvailabilityTracker implements Observer { + update(String status) } class PricingSystem implements Observer { + update(String status) } Car o--> Observer @enduml

			import java.util.ArrayList;
			import java.util.List;
		   
			interface Observer {
				void update(String status);
			}
	
	
			class Car {
				private String status;
				private List observers;
			
				public Car() {
					this.observers = new ArrayList<>();
				}
			
				public void addObserver(Observer observer) {
					observers.add(observer);
				}
			
				public void removeObserver(Observer observer) {
					observers.remove(observer);
				}
			
				public void setStatus(String status) {
					this.status = status;
					notifyObservers();
				}
			
				public void notifyObservers() {
					for (Observer observer : observers) {
						observer.update(status);
					}
				}
			}
			class CustomerNotification implements Observer {
				@Override
				public void update(String status) {
					System.out.println("CustomerNotification: The car status changed to: " + status);
				}
			}
			
			class AvailabilityTracker implements Observer {
				@Override
				public void update(String status) {
					System.out.println("AvailabilityTracker: The car is now: " + status);
				}
			}
			
			class PricingSystem implements Observer {
				@Override
				public void update(String status) {
					System.out.println("PricingSystem: Adjusting prices due to status change: " + status);
				}
			}

10.2 Chain of Responsibility

Jakub Sikora

Problem

Managing a vehicle reservation system involves handling a sequence of tasks such as vehicle selection, entering user details, and payment validation. Implementing this as a single monolithic function can lead to unreadable and unmaintainable code. Additionally, introducing new steps or modifying the existing ones can become cumbersome without a clear separation of concerns.

Solution

The Chain of Responsibility pattern allows us to encapsulate each step of the reservation process into a separate handler class. Each handler processes the request or delegates it to the next handler in the chain. This approach ensures flexibility, reusability, and maintainability of the code.

Class diagram

@startuml interface ReservationStep { + setNext(ReservationStep next) : void + handle(ReservationRequest request) : void } class AbstractReservationStep implements ReservationStep { - nextStep : ReservationStep + setNext(ReservationStep next) : void + handle(ReservationRequest request) : void } class VehicleSelectionStep extends AbstractReservationStep { + handle(ReservationRequest request) : void } class UserDataStep extends AbstractReservationStep { + handle(ReservationRequest request) : void } class PaymentProcessingStep extends AbstractReservationStep { + handle(ReservationRequest request) : void } class ReservationRequest { - selectedVehicle : String - userData : Map - paymentDetails : String + getSelectedVehicle() : String + setSelectedVehicle(String vehicle) : void + getUserData() : Map + setUserData(Map data) : void + getPaymentDetails() : String + setPaymentDetails(String details) : void } @enduml

			import java.util.ArrayList;
			import java.util.List;
			import java.util.Map;
		   
			public interface ReservationStep {
				void setNext(ReservationStep next);
				void handle(ReservationRequest request);
			}
	
	
			public abstract class AbstractReservationStep implements ReservationStep {
				private ReservationStep nextStep;
			
				@Override
				public void setNext(ReservationStep next) {
					this.nextStep = next;
				}
			
				protected void next(ReservationRequest request) {
					if (nextStep != null) {
						nextStep.handle(request);
					}
				}
			}

			public class VehicleSelectionStep extends AbstractReservationStep {
				@Override
				public void handle(ReservationRequest request) {
					if (request.getSelectedVehicle() == null || request.getSelectedVehicle().isEmpty()) {
						System.out.println("Vehicle selection is required.");
						return;
					}
					System.out.println("Vehicle selected: " + request.getSelectedVehicle());
					next(request);
				}
			}

			
			public class UserDataStep extends AbstractReservationStep {
				@Override
				public void handle(ReservationRequest request) {
					if (request.getUserData() == null || request.getUserData().isEmpty()) {
						System.out.println("User data is required.");
						return;
					}
					System.out.println("User data provided: " + request.getUserData());
					next(request);
				}
			}
			
			public class PaymentProcessingStep extends AbstractReservationStep {
				@Override
				public void handle(ReservationRequest request) {
					if (request.getPaymentDetails() == null || request.getPaymentDetails().isEmpty()) {
						System.out.println("Payment details are missing.");
						return;
					}
					System.out.println("Payment processed for details: " + request.getPaymentDetails());
					System.out.println("Reservation completed successfully!");
				}
			}

				}

					this.userData = userData;
				}
			
				public String getPaymentDetails() {
					return paymentDetails;
				}

				}
			}
